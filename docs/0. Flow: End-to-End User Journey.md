End-to-End User Journey – SAVI Portal

This document stitches together the core flows for the SAVI Portal, from first visit through authentication, scope resolution, navigation between platform and tenant, account/profile, invitations, and error/session handling. It is meant as the “master” journey that the other flow docs plug into.

Referenced routes and helpers are implemented in:
- `src/app` – route files and layouts
- `src/providers` – `AuthProvider`, `ScopeProvider`, `SessionProvider`
- `src/lib/auth` – guards, scope utilities, request context
- `src/components/layout` – shell, scope dropdown, user menu

⸻

1. First Visit → Authentication → Landing

1.1 Entry via `/`

Route: `src/app/page.tsx`

Flow:
1. User visits `/` (root).
2. `RootPage` reads auth state from `useAuthStore`:
   - If `status === 'loading'`: show `PageLoader` (“Loading SAVI...”).
   - Else if `status === 'authenticated'` and `profile` exists:
     - Calls `getLandingRoute()` from `AuthProvider` (internally uses `getDefaultLandingRoute(profile)`).
     - Hard-redirect via `router.replace(landingRoute)`.
   - Else:
     - Hard-redirect to login: `router.replace(ROUTES.LOGIN)` → `/login`.

Result: Anonymous users are always pushed to `/login`; authenticated users are sent directly to their default dashboard.

1.2 Login (`/login`)

Route: `src/app/(auth)/login/page.tsx`

Happy path:
1. User opens `/login`.
2. Enters email/password and submits form.
3. Page calls `signInWithEmail(email, password)` (Firebase auth).
4. On success, Firebase auth state changes – no manual redirect happens here.
5. `AuthProvider` (see 2.1 below) is subscribed to Firebase and:
   - Maps `User` → `AuthUser` (id, email, display name, idToken).
   - Calls `/api/v1/platform/auth/me` via `httpClient` using `fetchProfileWithScope` logic.
   - Updates `useAuthStore` with `{ status: 'authenticated', user, profile }`.
6. Login page `useEffect` sees `status === 'authenticated' && profile`:
   - Calls `getDefaultLandingRoute(profile)`.
   - `router.replace(landingRoute)` (e.g. `/platform/dashboard` or `/tenant/{slug}/dashboard`).

Failure path:
- On Firebase auth error:
  - `getFirebaseErrorMessage(error.code)` is shown inline above the form.
  - Form remains; user can retry.

1.3 Password Reset (`/reset-password`)

Route: `src/app/(auth)/reset-password/page.tsx`

Flow:
1. User opens `/reset-password` from login (“Forgot password?” link) or directly.
2. Enters email and submits.
3. Page calls `sendPasswordReset(email)` (Firebase).
4. On success:
   - Shows success state: “We sent a reset link to {email}”.
   - Offers “Back to sign in” (`/login`).
5. On error:
   - Maps error code via `getFirebaseErrorMessage` and shows inline error.

1.4 Landing Route Resolution

Decision logic lives in:
- `AuthProvider.getLandingRoute()` → `getDefaultLandingRoute(profile)`
- `src/lib/auth/guards.ts` → `getDefaultRoute(context)`
- `src/lib/auth/scope.ts` → `getLastScope()` / `saveLastScope()`

High-level rules:
1. Platform Admin:
   - Has a platform admin role (`isPlatformAdmin(profile) === true`).
   - Default landing: `/platform/dashboard`.
2. Tenant-only user:
   - No platform admin role, one or more `tenantMemberships`.
   - Default landing: tenant dashboard for either:
     - Last-used scope (if saved in `localStorage`), or
     - First membership as fallback.
   - URL: `/tenant/{tenantSlug}/dashboard`.
3. No platform access and no tenant memberships:
   - Landing: `/no-access`.

This logic is used:
- Immediately after login (in `LoginPage`).
- When computing “Go to my dashboard” from Unauthorized/404.

2. Global Providers & Scope Management

2.1 Provider Stack (Root Layout)

Route: `src/app/layout.tsx`  
Providers: `src/providers/AppProviders.tsx`

Hierarchy:
1. `<ToastProvider>` – global notifications.
2. `<AuthProvider>` – Firebase auth + `/auth/me` + auth store.
3. `<ScopeProvider>` – platform vs tenant scope (tenantId, slug).
4. `<SessionProvider>` – session expiry dialog on 401s.

Effect: Every page in the app sees a consistent auth profile, scope context, and session handling.

2.2 AuthProvider – Firebase & `/auth/me`

File: `src/providers/AuthProvider.tsx`

Responsibilities:
1. Subscribe to Firebase auth state (`subscribeToAuthState`).
2. Configure `httpClient` with token getter (`setTokenGetter(getIdToken)`).
3. On user sign-in:
   - Map Firebase `User` → `AuthUser`.
   - Call `fetchProfileWithScope()`:
     - Checks current URL:
       - `/tenant/{slug}`: tries cached tenant (`tenant-cache`); may call `/auth/me` once with `tenantId`, or twice (initial + tenant-scoped).
       - `/platform/...`: sets platform scope and calls `/auth/me`.
       - Other paths: single `/auth/me`.
     - Caches all tenant memberships for future lookups.
   - Calls `setAuthenticated(user, profileData)` in `useAuthStore`.
4. On sign-out:
   - Clears auth store and redirects to `/login` (see 6.2).
5. Exposes helpers via context:
   - `logout()`, `refreshProfile()`, `getLandingRoute()`.

2.3 ScopeProvider – Platform vs Tenant

File: `src/providers/ScopeProvider.tsx`

Responsibilities:
1. Sync scope with URL:
   - Uses `usePathname()` and `extractTenantSlug(pathname)`.
   - If tenant slug present and `profile` loaded:
     - Finds matching `tenantMembership`.
     - Calls `setTenantScope(tenantId, tenantSlug)` in `useScopeStore`.
   - If path starts with `/platform`:
     - Calls `setPlatformScope()`.
2. Configure `httpClient` with tenant ID:
   - `setTenantIdGetter(() => tenantId)` so all tenant calls send correct header.
3. Handle tenant switch:
   - Tracks initial tenant ID from `AuthProvider`.
   - When `tenantId` changes later (scope switch), calls:
     - `/auth/me` with the new `tenantId`.
     - Updates `profile` in `useAuthStore`.
4. Exposes:
   - `refreshProfileForScope()` – manual refresh.
   - `getCurrentTenantId()` – used by modules for direct access.

2.4 SessionProvider – Session Expiry

File: `src/providers/SessionProvider.tsx`

Responsibilities:
1. Registers a session-expired handler with `httpClient`:
   - `setSessionExpiredHandler(() => setSessionExpired(true))`.
2. When `isSessionExpired` is true in `useSessionStore`, renders:
   - `<SessionExpiredDialog />`.

Session flow is detailed in section 6.3.

3. Shell & Navigation

3.1 Core Shell

File: `src/components/layout/AppShell.tsx`

Structure:
1. Sidebar (`SideNav`):
   - Desktop fixed at left.
   - Mobile slide-in with overlay.
   - Items adapt to current `scope` and `profile.permissions`.
2. Top bar (`TopBar`):
   - Hamburger for mobile sidebar.
   - Scope dropdown (`ScopeDropdown`).
   - User menu (`UserMenu`).
3. Main content:
   - Wraps page content with consistent padding and background.

3.2 Scope Dropdown (Platform vs Tenant)

File: `src/components/layout/ScopeDropdown.tsx`

Behaviour:
1. Builds options via `buildScopeOptions(profile)`:
   - Platform admins:
     - `Platform` + all tenant memberships.
   - Tenant-only users:
     - All tenant memberships (no platform option).
2. Infers current scope from URL using `getCurrentScopeFromPath(pathname)`.
3. On change:
   - Finds selected `ScopeOption`.
   - Saves it as “last used” via `saveLastScope(scope)`.
   - Hard-navigates using `window.location.href` to:
     - `/platform/dashboard` for platform scope.
     - `/tenant/{tenantSlug}/dashboard` for tenant scope.
4. Full reload ensures:
   - `ScopeProvider` picks up new scope from URL.
   - `/auth/me` is re-fetched with correct `tenantId` header.
   - Sidebar and permissions update accordingly.

3.3 User Menu (Profile & Logout)

File: `src/components/layout/UserMenu.tsx`

Behaviour:
1. Shows avatar + name derived from:
   - `profile.displayName` or `user.displayName`.
   - Email from `profile.email` or `user.email`.
2. Dropdown actions:
   - “My Profile”:
     - `router.push(ROUTES.ACCOUNT.PROFILE)` (`/account/profile`).
   - “Sign Out”:
     - Calls `logout()` from `AuthProvider` (see 6.2).

4. Guarded Layouts & Main Scopes

4.1 Platform Layout

Route: `src/app/(platform)/platform/layout.tsx`

Guard logic:
1. Reads `status` and `profile` from `useAuthStore`.
2. Builds `RequestContext` from profile + pathname (`buildRequestContext`).
3. Calls `requirePlatformAccess(context)`:
   - If not authenticated:
     - Saves `redirectAfterLogin` in `sessionStorage`.
     - Redirects to `/login`.
   - If authenticated but not platform admin:
     - Redirects to `/unauthorized`.
   - If allowed:
     - Renders `AppShell` with `children`.
4. While checking:
   - If `status === 'loading'`: shows `PageLoader` (“Loading platform...”).
   - If not authorized yet: shows `PageLoader` (“Verifying access...”).

4.2 Platform Dashboard

Route: `src/app/(platform)/platform/dashboard/page.tsx`

Main elements:
1. Header:
   - “Platform Dashboard”.
   - Greets Platform Admin using `profile.displayName`.
2. Stats cards:
   - Total communities (`tenantMemberships.length`).
   - Placeholder metrics: total users, active sessions, system health.
3. Communities list:
   - For the first few `tenantMemberships`:
     - Shows name + slug.
     - “Open →” link to `/tenant/{slug}/dashboard`.
4. Quick actions:
   - Manage communities → `/platform/tenants`.
   - Manage users → `/platform/users` (placeholder route).

4.3 Tenants List (Platform)

Route: `src/app/(platform)/platform/tenants/page.tsx`

Key behaviours:
1. Uses `profile.permissions` to derive:
   - `canView` → `PLATFORM_TENANT_VIEW`.
   - `canManage` → `PLATFORM_TENANT_MANAGE`.
   - `canUpdate` → `PLATFORM_TENANT_UPDATE`.
2. Fetches tenants via `listTenants({ page, pageSize, status, search })`.
3. Provides:
   - Search, status filters, pagination.
   - Actions per tenant:
     - View details → `/platform/tenants/{tenantId}`.
     - Edit tenant (if `canUpdate`).
     - Invite admin (if `canManage`).
     - Archive (if `canManage`, includes confirmation and `archiveTenant` call).
4. Error and empty states:
   - Loading spinner while fetching.
   - Error state with retry if fetch fails.
   - Empty state with CTA to create first community if allowed.

4.4 Tenant Layout

Route: `src/app/(tenant)/tenant/[tenantSlug]/layout.tsx`

Guard logic:
1. Reads `status` and `profile` from `useAuthStore`.
2. Extracts `tenantSlug` from URL (`useParams`).
3. Builds `RequestContext` and calls `requireTenantAccess(context, tenantSlug)`:
   - If not authenticated:
     - Saves `redirectAfterLogin` and redirects to `/login`.
   - If no membership for `tenantSlug`:
     - Redirects to `/unauthorized`.
   - If allowed:
     - Renders `AppShell` with `children`.
4. Additionally:
   - Maintains `showNotFound` flag. If set, calls `notFound()` which triggers global 404.
5. While checking:
   - If `status === 'loading'`: show “Loading community...” loader.
   - If not yet authorized: show “Verifying access...” loader.

4.5 Tenant Dashboard

Route: `src/app/(tenant)/tenant/[tenantSlug]/dashboard/page.tsx`

Behaviour:
1. Reads `tenantSlug` from `useParams()`.
2. Resolves membership via `hasAccessToTenant(profile, tenantSlug)`:
   - Uses membership to derive tenant name and user roles.
3. Renders:
   - Header:
     - Tenant name (fallback to slug).
     - Role badges for each tenant role.
   - Stats cards:
     - Total units, residents, parking slots, open maintenance (placeholder values).
   - Sections:
     - Recent activity placeholder.
     - Quick actions:
       - Manage units → `/tenant/{slug}/units`.
       - Manage residents → `/tenant/{slug}/residents`.
       - Maintenance → `/tenant/{slug}/maintenance`.

4.6 Account Layout & Profile

Account layout: `src/app/account/layout.tsx`  
Profile route: `/account/profile` (page file not included here but flows defined in `docs/2. Flow: Open & View My Profile.md`)

Account layout guard:
1. Builds `RequestContext` and calls `requireAuthenticated(context)`:
   - Not authenticated:
     - Saves `redirectAfterLogin`.
     - Redirects to `/login`.
   - Authenticated:
     - Renders `AppShell` with account pages (e.g. profile).
2. While checking:
   - `status === 'loading'` → generic loader.
   - Not yet authorized → “Verifying access...” loader.

Profile flow:
- From anywhere in the shell, user opens user menu → “My Profile”:
  - Navigates to `/account/profile`.
  - Shell (scope dropdown + sidebar) still reflects the current scope.
  - Profile content shows personal info + roles & tenant memberships.
  - Update behaviour is described in `docs/2. Flow: Open & View My Profile.md`.

5. Invitations & First-Time Tenant Access

5.1 Accept Invitation

Route: `src/app/(auth)/invitations/accept/page.tsx`  
URL: `/invitations/accept?token=...`

Flow:
1. On mount, reads `token` from query string.
2. Validates token:
   - Calls `validateInvitation(token)`.
   - On success:
     - Stores `invitation` details (tenant name, city, role, expiry, invitee email).
     - Sets step to `details`.
   - On failure:
     - Sets `step = 'invalid'` and shows error state:
       - “Invalid invitation” message.
       - “Go to Login” button → `/login`.
3. From details step:
   - Shows invitation summary (community, location, role, expiry).
   - Shows locked email address (inviteeEmail).
   - “Continue” button moves to `step = 'signin'`.
4. Sign-in / sign-up:
   - User chooses:
     - Existing account (default).
     - “I don’t have an account yet” (checkbox).
   - For new user:
     - Validates password match and length (>= 6).
     - Calls `createUserWithEmail(invitation.inviteeEmail, password)`.
   - For existing user:
     - Calls `signInWithEmail(invitation.inviteeEmail, password)`.
   - On Firebase errors:
     - Uses `getFirebaseErrorMessage(err.code)` and stays in `signin` step.
5. Accepting invitation:
   - On successful sign-in/up:
     - Sets step `accepting`.
     - Calls `acceptInvitation(token)` API.
     - Result includes `{ tenantId, tenantCode }`.
     - Calls `setTenantScope(tenantId, tenantCode)` in `useScopeStore`.
     - Calls `fetchAuthMe()` to refresh profile with new tenant membership.
     - Sets step `success` and shows welcome message.
     - After short delay, navigates to `/tenant/{tenantCode}/dashboard`.

5.2 Resulting Journey for Invitee

First-time invited user:
1. Clicks invite link → `/invitations/accept?token=...`.
2. Valid token:
   - Sees community + role info, locked email.
3. Creates account or signs in with the invited email.
4. Invitation is accepted; profile now contains tenant membership.
5. Lands on tenant dashboard for that community.

6. Error, Unauthorized & Session Flows

6.1 Global Error

Route: `src/app/error.tsx`

Behaviour:
1. Catches render errors at the app level (Next.js error boundary).
2. Logs error to console for debugging.
3. Shows:
   - Generic “Something went wrong” message.
   - In development: error message and optional digest.
4. Actions:
   - “Try again” (calls `reset()` from Next error boundary).
   - “Go home” (hard redirect to `/`).

6.2 Unauthorized & No Access

Unauthorized page: `src/app/unauthorized/page.tsx`  
No access page: `src/app/no-access/page.tsx`

Unauthorized (`/unauthorized`):
1. Shown when guards redirect due to insufficient permissions:
   - Non-platform-admin hitting `/platform/...`.
   - User trying to access a tenant they are not a member of.
2. If `profile` exists:
   - Shows email and global roles.
   - Computes `defaultRoute` using `getDefaultRoute(buildRequestContext(profile, '/unauthorized'))`.
3. Actions:
   - “Go to my dashboard” → navigates to default route.
   - “Go back” → `window.history.back()`.

No access (`/no-access`):
1. Shown when user is authenticated but has:
   - No platform admin role.
   - No tenant memberships.
2. Explains that the account has no assigned communities or platform areas.
3. Shows signed-in email.
4. Offers:
   - Guidance to contact administrator.
   - “Sign out and try another account” (calls `logout()` from `AuthProvider`).

6.3 Not Found (404)

Route: `src/app/not-found.tsx`

Behaviour:
1. Used for:
   - Non-existent routes at app level.
   - Tenant-level not found via `notFound()` from `TenantLayout`.
2. Uses `useAuthStore`:
   - If authenticated and `profile` exists:
     - Computes `dashboardLink` with `getDefaultRoute(buildRequestContext(profile, '/not-found'))`.
   - Else:
     - Sets `dashboardLink` to `/login`.
3. UI:
   - 404 illustration.
   - Buttons:
     - Primary: “Back to dashboard” (or “Go to login” when unauthenticated).
     - Secondary: “Go back” (history.back).

6.4 Session Expiry

Files:
- `src/providers/SessionProvider.tsx`
- `src/components/feedback/SessionExpiredDialog.tsx`

Trigger:
- Any API call that results in 401, via `httpClient`, calls the registered session expired handler.
- `useSessionStore` sets `isSessionExpired = true`.

Dialog behaviour:
1. “Continue Session”:
   - Calls `getIdToken(true)` to force refresh Firebase token.
   - On success:
     - Clears `isSessionExpired`.
     - Reloads the page (`window.location.reload()`).
   - On failure:
     - Shows error: “Unable to refresh session. Please sign in again.”
2. “Sign Out”:
   - Calls `firebaseSignOut`.
   - Clears last scope (`clearLastScope`).
   - Hard redirects to `/login`.

7. Summary: High-Level Journey by Persona

7.1 Platform Admin

1. Visit `/` → redirected to `/login` if not authenticated.
2. Log in:
   - `/auth/me` shows platform admin role and tenant memberships.
   - Landing route: `/platform/dashboard`.
3. Uses scope dropdown to:
   - Stay in Platform scope (manage communities, users, platform RBAC).
   - Switch into specific tenant(s) as needed:
     - Hard navigate to `/tenant/{slug}/dashboard`.
4. From any scope:
   - Opens `/account/profile` via user menu.
   - Signs out via user menu; redirected to `/login`.

7.2 Tenant Admin / Community Manager

1. Visit `/` → `/login` if not authenticated.
2. Log in:
   - `/auth/me` returns tenant memberships but no platform admin role.
   - Landing route: `/tenant/{slug}/dashboard` (default or last-used tenant).
3. Uses scope dropdown to:
   - Switch between multiple communities they manage.
4. In tenant scope:
   - Uses tenant modules (units, parties, ownership, leases, residents, RBAC) as defined in:
     - `docs/3.Flow:Community Setup.md`
     - `docs/3. Flow Party.md`
     - `docs/4.flow-rbac.md`
     - `docs/5.flow-resident.md`
5. Accesses `/account/profile` and signs out the same way as Platform Admin.

7.3 Invited Tenant Admin (First-Time)

1. Receives invite and clicks `/invitations/accept?token=...`.
2. Valid token:
   - Reviews invitation details with locked email.
3. Creates account or signs in with invited email.
4. Invitation is accepted, tenant membership added to profile.
5. Lands on `/tenant/{tenantCode}/dashboard`.

This file should now be the single starting point for understanding the overall user journey through the SAVI Portal; the other flow docs plug into specific areas (profile, units, parties, RBAC, residents) referenced above.
