🌟 UNIT-CENTRIC FLOWS

📘 F-UNIT-LIST-01 – View Units List

Actor: Tenant user with TENANT_STRUCTURE_VIEW
URL: /tenant/{slug}/units

Purpose: This is the central entry point into the community structure.

Steps
	1.	User navigates to /tenant/{slug}/units:
	•	Via side nav → “Units”
	•	Or from dashboard card → “View all units”
	2.	Tenant layout guard:
	•	Confirms tenant membership
	•	Confirms permission TENANT_STRUCTURE_VIEW
	3.	Page loads with filters:
	•	Block dropdown (auto-populated from /blocks)
	•	Floor dropdown (dependent on block)
	•	Unit Type dropdown
	•	Status dropdown (active/inactive, vacant/occupied)
	4.	Frontend calls:

GET /api/v1/tenant/community/units?blockId=&floorId=&status=&type=


	5.	Units table shown with:
	•	Unit Number
	•	Block Name
	•	Floor Name
	•	Unit Type
	•	Occupancy status (Vacant/Occupied)
	•	Parking slots count (if pre-fetched)
	6.	Actions available:
	•	New Unit (if TENANT_STRUCTURE_MANAGE)
	•	Open (always allowed if view permission)

Error Handling
	•	403 → “You don’t have access to units.”
	•	401 → Session expiry dialog
	•	404 → Only if tenant slug invalid → global not-found handler

📘 F-UNIT-DETAIL-01 – Open Unit Detail (Hub Screen)

Actor: Tenant user with TENANT_STRUCTURE_VIEW
URL: /tenant/{slug}/units/{unitId}

Purpose:
This is the core control center for the entire structural hierarchy.

Steps
	1.	User clicks a unit row in F-UNIT-LIST-01.
	2.	App calls:

GET /api/v1/tenant/community/units/{unitId}


	3.	Page renders:

A. Breadcrumb (top-down navigation)

Block A → Floor 3 → Unit 3A

Each is a link to:
	•	Block A → /tenant/{slug}/floors?blockId=X
	•	Floor 3 → /tenant/{slug}/units?blockId=X&floorId=Y
	•	Unit 3A (current)

B. Unit Info Panel

Fields from swagger/DB:
	•	Unit Number
	•	Unit Type
	•	Block
	•	Floor
	•	Area (sqft)
	•	Status (Active)
	•	Notes
	•	Attachments (if included)

Actions:
	•	Edit Unit (if TENANT_STRUCTURE_MANAGE)
	•	Mark Inactive/Active

C. Parking Allocation Section (bottom-up navigation)

Displays parking slots linked to this unit:
	•	Code (linkable)
	•	Level Label
	•	Is Covered / EV
	•	Parking Status
	•	Allocation Status

Actions:
	•	Add parking to this unit
→ opens Unit-centric allocation dialog
	•	Remove allocation
	•	Each parking slot row:
	•	Click → /tenant/{slug}/parking-slots/{slotId}
(Bottom → Up path)

D. Optional Expandable Sections (future):
	•	Residents assigned to the unit
	•	Maintenance history
	•	Documents

📘 F-UNIT-DETAIL-02 – Edit Unit

Actor: Tenant user with TENANT_STRUCTURE_MANAGE

Steps
	1.	On Unit Detail page → user clicks Edit
	2.	Dialog opens with editable fields from PUT /units/{id}:
	•	Block (dropdown)
	•	Floor (dropdown filtered by block)
	•	Unit Type
	•	Unit Number
	•	AreaSqft
	•	Status
	•	Notes
	3.	User submits →

PUT /api/v1/tenant/community/units/{unitId}


	4.	On success:
	•	Dialog closes
	•	Unit detail refreshes
	•	Show toast “Unit updated”

Errors
	•	409 → “Unit already has dependencies; you cannot change block/floor”
	•	403 → “No permission to modify units”
	•	401 → session expiry dialog

📘 F-UNIT-DELETE-01 – Delete Unit

Actor: Tenant user with TENANT_STRUCTURE_MANAGE

Steps
	1.	On Unit Detail → click “Delete Unit”
	2.	Confirm dialog:
	•	“This unit may have residents/parking/maintenance tied to it.”
	3.	Submit:

DELETE /api/v1/tenant/community/units/{unitId}


	4.	Outcomes:
	•	200/204 → Navigate back to /units, toast “Unit deleted”
	•	409 Conflict
	•	Show message:
“This unit is in use and cannot be deleted. Mark inactive instead.”

🌟 PARKING → UNIT → PARKING CYCLE (Two-way binding)

📘 F-ALLOCATE-PARKING-01 – Allocate parking from unit perspective
	1.	On Unit Detail → Parking section → “Add parking”
	2.	Dialog loads available slots:

GET /parking-slots?status=Available


	3.	Admin selects slot P-12 → click “Allocate”
	4.	API call:

POST /parking-slots/{id}/allocate
{
  "unitId": <unitId>
}


	5.	Unit detail refresh → P-12 appears in list

📘 F-ALLOCATE-PARKING-02 – Allocate from parking perspective
	1.	On Parking List → row → “Allocate”
	2.	Dialog: choose Unit
	3.	Backend call:

POST /parking-slots/{id}/allocate


	4.	UI navigates to Parking detail
	5.	Parking detail offers link: Open Unit 3A

🌟 PERMISSION & ERROR FLOWS

📘 F-PERMISSION-01 – Lack of structure manage permission

If user has only TENANT_STRUCTURE_VIEW:
	•	Cannot see:
	•	New Unit
	•	Edit Unit
	•	Delete Unit
	•	Add Parking
	•	Can still:
	•	View Unit list
	•	View Unit detail
	•	Navigate hierarchy
	•	View parking allocations

📘 F-ERROR-01 – Invalid Unit Id

If unit isn’t found:
	•	Backend returns 404
	•	Tenant layout triggers:
	•	Next.js notFound() → tenant-styled 404 page
“This unit doesn’t exist or you don’t have access.”

📘 F-ERROR-02 – Session Expired

Any API call returning 401 triggers:
	•	Global session dialog “Continue session / Sign out”
	•	On “Continue session”, retry allowed action if applicable
