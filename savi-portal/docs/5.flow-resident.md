1. Residents module – main screens

For Community Admin, the module has 3 key screens:
	1.	Residents List
	2.	Resident Profile
	3.	Resident Actions (flows triggered from profile or list)

We’ll define flows around these.


2. Flow 1 – Residents List (entry point)

Goal

Give Community Admin a single screen to see:
	•	All current residents
	•	Upcoming move-ins
	•	Past residents (for history)

Steps
	1.	Admin goes to Residents → List in side nav.
	2.	System builds the list from:
	•	LEASE_PARTY where Role in [PrimaryResident, CoResident]
	•	Join LEASE (only Status = Active or also historical if filtering)
	•	Join UNIT to show where they live.
	•	Join PARTY for name.
	•	Left-join COMMUNITY_USER to see if they have an account.
	3.	UI shows table:
	•	Resident name (from PARTY)
	•	Unit (Block/Floor/Unit)
	•	Status: Upcoming / Current / Past
	•	Primary / Co-Resident badge
	•	Has App Access? (Yes/No)
	•	Open tickets count (optional)
	4.	Filters / segments:
	•	Status (Upcoming / Current / Past)
	•	Unit / Block / Floor
	•	Has App Access (Yes/No)
	•	Text search by name/phone/email.
	5.	Actions per row:
	•	View profile
	•	Invite / Resend invite (if no CommunityUser yet)
	•	Move out (end residency)
	•	Add / manage co-residents for that unit

    3. Flow 2 – Resident Profile

Goal

See everything about one resident in one place.

Steps
	1.	From Residents List, admin clicks a resident → Resident Profile.
	2.	System loads:
	•	Party (identity & PII)
	•	CommunityUser (if any) → login status, last login
	•	LeaseParty records → all leases where they are resident
	•	Lease & Unit for each → current vs past unit(s)
	•	ResidentInvites (sent, pending, accepted)
	•	MaintenanceRequests they requested (requested_by / requested_for)
	•	Optionally ownership (if same Party is also an owner elsewhere)
	3.	Profile sections:
	•	Header: Name, Phone, Email, Current Unit, Status.
	•	Residency tab:
	•	Current lease (Unit, start/end, move-in date)
	•	Co-residents in same unit
	•	Past residencies (previous units)
	•	Account tab:
	•	Has CommunityUser? (Yes/No)
	•	Email/phone used to log in
	•	Invite status (Pending / Accepted / Expired)
	•	Actions: Send invite / Resend / Revoke
	•	Activity tab:
	•	Maintenance requests list
	•	Important timeline items (move in/out, lease changes, invites)
	4.	From here, admin can trigger key actions (see next flows).


    5. Flow 4 – Invite / Resend Invite

There are two entry points:
	•	From Residents List (row action: Invite/Resend)
	•	From Resident Profile → Account tab

Steps
	1.	Admin clicks Invite / Resend Invite.
	2.	System checks:
	•	Does this resident’s Party already have a CommunityUser?
	•	If yes: show message: “They already have an account” (no new invite needed).
	•	If no: create/renew RESIDENT_INVITE:
	•	LeaseId and PartyId from current active lease + lease_party.
	•	Role from LeaseParty (PrimaryResident/CoResident).
	•	Generate token, set Status = Pending.
	3.	System sends email/SMS with invite link.

Accepting the invite is exactly your existing flow3-residentflow.md:
	•	Resident clicks link → goes through signup → app creates COMMUNITY_USER pointing to this PARTY → marks invite Accepted.

    Flow 2 – Invite resident / co-resident to the app (from unit)

Actor: Community Admin
Start: Unit details → Residents list (derived from LeaseParty+Party)
	1.	For each resident (Primary or Co), show:
	•	Name
	•	Role (Primary / Co)
	•	App access: Yes/No (derived by checking CommunityUser for that Party).
	2.	For residents with no CommunityUser:
	•	Show “Invite to App” button.
	3.	When admin clicks “Invite to App”:
	•	System validates:
	•	There is an Active Lease.
	•	This Party is a LeaseParty with Role = PrimaryResident/CoResident.
	•	System creates a ResidentInvite:
	•	LeaseId = Active Lease.Id
	•	PartyId = LeaseParty.PartyId
	•	Role = LeaseParty.Role
	•	CreatedByUserId = admin CommunityUser.Id
	•	Status = Pending
	•	Token and contact details (email/phone)
	4.	System sends email/SMS with the invite link.
	5.	Resident clicks link → goes through flow3-residentflow:
	•	Signs up → CommunityUser created for that Party.
	•	Invite marked Accepted.

✅ This is now clearly: Unit → Lease → LeaseParty → ResidentInvite, 

Flow 3 – Add resident starting from Residents List (must route via unit/lease)

We keep your rule: you must still pick a unit with an Active Lease; you cannot spawn a new lease here.

Actor: Community Admin
Start: Residents List screen → “Add Resident” button OR “Assign to Unit” action for a Party.
	1.	Admin clicks “Add Resident” on Residents List.
	2.	System asks:
	•	Either:
	•	Select an existing Party (if starting from generic “Add Resident”), or
	•	If action came from a specific Party row → Party is already known.
	3.	System then asks:
	•	“Select a Unit (with Active Lease) for this resident.”
	•	Unit picker only shows units with Active Lease (or clearly marks ones without and disables them).
	4.	Once admin selects a unit, the system:
	•	Redirects to the Unit’s “Manage Occupancy” view, with this Party pre-selected.
	•	Then it runs Flow 1.2:
	•	Add them as Primary or CoResident depending on existing primary.
	5.	Before final save, system checks:
	•	Party is not already a current resident under another Active Lease.
	•	If they are → warn / block / require explicit override, as per your rule.

✅ So Residents List never creates a lease and never creates “current resident without unit”.
It’s just a starting shortcut into the Unit-centric occupancy flow.


Flow 4 – Move out resident (always from unit)

You’re right: move-out should be done from the Unit, because it’s about freeing a unit or changing occupancy of that unit’s lease.

Actor: Community Admin
Start: Unit details → Residents list under Active Lease
	1.	Admin clicks “Move Out” next to a resident row (LeaseParty).
	2.	System knows:
	•	Unit
	•	Active Lease Id
	•	LeaseParty row (role, IsPrimary, etc.)
	3.	System asks for Move-out date and offers options depending on role:
	•	If CoResident:
	•	Just set LeaseParty.MoveOutDate = chosen date.
	•	They become Past Resident for that unit.
	•	If PrimaryResident:
	•	UI asks:
	1.	“End entire lease?” →
	•	If yes:
	•	Set Lease.Status = Ended (and maybe EndDate).
	•	Set all LeaseParty.MoveOutDate accordingly.
	2.	“Or keep lease active and make someone else primary?” →
	•	Show list of other active LeaseParties.
	•	Admin chooses a new primary → update IsPrimary flags.
	•	Set MoveOutDate on the old primary.
	4.	After move out:
	•	If lease still Active and other residents exist → unit remains occupied.
	•	If lease ended (no active lease) → unit becomes vacant, all its residents appear under Past in Residents List.


    Flow 5 – Manage co-residents (this is just a slice of Flow 1)

No separate fancy flow; it’s literally just “Add/Remove residents on this unit’s active lease”.

Actor: Community Admin
Start: Unit details → Residents list
	1.	To add co-resident:
	•	Click “Add Resident”.
	•	If there is already a primary:
	•	Any new LeaseParty created will default to Role = CoResident, IsPrimary = false.
	•	Same Party selection and validation rules as Flow 1.2.
	2.	To remove co-resident:
	•	Click “Move Out” next to co-resident → same as Flow 4 for CoResident branch.


    1.2 Add resident to an active lease (primary or co-resident)

Actor: Community Admin
Start: Unit details → Active Lease visible
	1.	Admin clicks “Add Resident” within the “Residents for this Lease” section.
	2.	System ensures:
	•	There is an Active Lease for this unit.
	3.	Admin selects / creates Party:
	•	Search Party by name/phone/email OR
	•	Create new Party with minimal data.
	4.	System checks:
	•	Is this Party already a current resident in another unit (other Active Lease)?
	•	If yes, show warning:
“This person is already a resident of Unit XYZ on an active lease.”
Depending on business rule:
	•	Either block or
	•	Allow with explicit override.
	5.	System sets LeaseParty:
	•	If there is no primary resident on this lease:
	•	Make this Party → Role = PrimaryResident, IsPrimary = true.
	•	If there is already a PrimaryResident:
	•	Make this Party → Role = CoResident, IsPrimary = false.
	6.	Save LeaseParty with:
	•	LeaseId = Active Lease.Id
	•	PartyId = selected Party.Id
	•	MoveInDate (default to Lease.StartDate or today)
	•	IsPrimary as decided.
	7.	After save, unit’s “Residents” list updates.
	8.	Optional next step: UI shows “Invite to App” button next to any Party without CommunityUser 

    Core rules (so we don’t violate your intent)
	1.	All current residents must come from an Active Lease.
	•	You cannot create a new Lease from “Add Resident”.
	•	You must start from Lease (or Unit → Lease), then add residents to that lease.
	2.	No “floating current residents” without a unit/lease.
	•	A Party is only a current resident if:
	•	They are on a LeaseParty with Role in [PrimaryResident, CoResident]
	•	The Lease is Active and their MoveOutDate is null / in future.
	•	If the lease expired or they moved out → they become a Past Resident.
	3.	Everything operational is Unit-centric.
	•	Add resident → from Unit with Active Lease.
	•	Move out → from Unit (not from Resident).
	•	Manage co-residents → from Unit (not from Resident).
	4.	Residents List is mostly a lookup / navigation tool.
	•	Heavy actions always bounce back to Unit → Lease.

This is the one main flow that covers:
	•	Add primary resident
	•	Add co-resident
	•	Invite to app
	•	Move out
	•	(Optional) End / renew lease

From Unit details:
	•	Show:

	•	Current residents (LeaseParty rows)
	•	If no Active Lease:
	•	Show: “No active lease for this unit. Create a lease first.”
	•	Action: Create Lease (goes to Lease flow, separate spec).
