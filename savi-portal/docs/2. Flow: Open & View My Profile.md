1. High-level concept / approach

1.1 Where profile lives
	•	Top-right user menu in the global header, visible in all scopes:
	•	Shows avatar (or initials) + short name
	•	On click → dropdown with:
	•	My profile
	•	Sign out

This is the same for:
	•	Platform Admin on /platform/...
	•	Tenant Admin on /tenant/{slug}/...

1.2 Route for profile

Use a single profile route for both personas:
	•	/account/profile (or /profile — naming detail for later)
	•	Layout shell stays the same:
	•	Top bar (scope dropdown + user menu)
	•	Left navigation (based on current scope: platform nav or tenant nav)
	•	Main content area shows the profile screen, which is not scope-specific:
	•	Your personal info
	•	Overview of roles & access
	•	Notification preferences
	•	Security links (manage password, 2FA) — delegated to auth provider if needed

So if user is in Sunrise Heights and opens Profile:
	•	URL: /account/profile
	•	Scope dropdown still shows Sunrise Heights
	•	Left nav still shows tenant nav
	•	Main content: profile

If they switch scope while on profile:
	•	URL stays /account/profile
	•	Only role/access summary part changes (because tenant membership list might be highlighted per scope, but the profile route itself is global).

1.3 What profile shows (foundation)

Think of this as production-ready but extensible:
	1.	Section 1 – Basics
	•	Display name
	•	Primary email (usually read-only if managed by Firebase/OIDC)
	•	Phone number (editable)
	•	Timezone (optional, or default to tenant/platform)
	•	Avatar upload/change
	2.	Section 2 – Access & roles overview
	•	Global roles:
	•	Example: PlatformAdmin, PlatformSupport
	•	Tenant memberships list:
	•	Tenant name + slug
	•	Roles in that tenant: TenantAdmin, CommunityManager, etc.
	•	This helps users understand “why do I see these options?”
	3.	Section 3 – Notifications
	•	Email notifications toggles (for now basic):
	•	Maintenance updates
	•	Announcements
	•	Marketplace
	•	Can be refined later, but the structure is in place.
	4.	Section 4 – Security (integrated with auth provider)
	•	Buttons that send user to:
	•	“Change password” (link to Firebase password reset flow or IdP)
	•	“Manage 2FA” (if you support it later)
	•	In app we show last login info (read-only):
	•	Last login date/time
	•	Logged-in device/browser

1.4 Logout behavior

Logout is simple and global:
	•	Available in user dropdown as “Sign out”.
	•	Behavior:
	1.	Call auth provider’s sign-out (firebase.auth().signOut() or equivalent).
	2.	Clear local storage / cookies (tokens, cached /auth/me, last scope).
	3.	Hard redirect to /login.
	•	After logout, user always starts afresh:
	•	No remembered scope.
	•	No cached permissions.

This is true whether they clicked it from /platform/... or /tenant/....

3. Flow: Update Profile Basics (Name, Phone, Avatar)

3.1 User Journey – Update basic info

Name: F-PROFILE-02

Goal: User updates editable profile fields (e.g. display name, phone, avatar).

Actors:
	•	Platform Admin
	•	Tenant Admin

Preconditions:
	•	User is on /account/profile.
	•	Backend has an endpoint for profile updates, e.g. PUT /account/profile.

Steps:
	1.	On /account/profile, user sees Basics form with:
	•	Display name (editable)
	•	Phone (editable)
	•	Email (read-only)
	•	Avatar upload/change button
	2.	User modifies one or more fields.
	3.	Save button highlighted/enabled.
	4.	User clicks “Save changes”.
	5.	Frontend sends request to PUT /account/profile.
	6.	Backend:
	•	Validates data.
	•	Updates user record in Platform DB.
	•	Returns updated profile.
	7.	Frontend:
	•	Updates local profile state.
	•	Shows success toast: “Profile updated.”
	8.	Subsequent /auth/me or profile loads show updated values.

Optional error flow:
	•	If validation fails: show inline errors.
	•	If network error: show error toast and keep form as-is.

4. Flow: Logout / Sign Out

4.1 User Journey – Logout from any scope

Name: F-LOGOUT-01

Goal: User signs out cleanly and is brought back to the login page.

Actors:
	•	Any authenticated user (Platform Admin or Tenant Admin).

Preconditions:
	•	User is authenticated and currently on /platform/... or /tenant/{slug}/....

Steps:
	1.	User clicks avatar/name in top-right header.
	2.	User menu dropdown appears.
	3.	User selects “Sign out”.
	4.	App triggers logout flow:
	•	Call auth provider’s sign-out (e.g. firebase.auth().signOut()).
	•	Call backend logout endpoint if you have one (for server-side sessions / audit).
	•	Clear local tokens, cached /auth/me, and “last used scope” from storage.
	5.	Once logout is confirmed/successful:
	•	Hard redirect to /login (window.location.href = "/login").
	6.	/login page loads with no authenticated context.

Post-condition:
	•	User is fully signed out.
	•	Any previous tenant or platform context is dropped.
	•	Next action requires login again.
