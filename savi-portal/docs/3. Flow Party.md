Here’s how I’d shape the Tenant → Parties module in the tenant web app, wiring it to those APIs and permissions.

⸻

1. Tenant View – Screens & URLs

Scope: Tenant app, inside /tenant/{tenantSlug} shell.
Permissions:
	•	TENANT_PARTY_VIEW – can see lists & details
	•	TENANT_PARTY_MANAGE – can create/update/delete parties, addresses, contacts

Main route:
	•	/tenant/{tenantSlug}/parties

1.1 Parties List Screen

URL: /tenant/{tenantSlug}/parties

Contents:
	•	Header: Parties
	•	Controls:
	•	Search (by name, code, email/phone)
	•	Filters (Type: Person / Organization / Vendor / Resident / Owner, etc.)
	•	New Party button – visible only if TENANT_PARTY_MANAGE.
	•	Table columns (example, mapped from Party):
	•	Name
	•	Type (Person/Org)
	•	Category (Owner / Resident / Vendor / Other)
	•	Primary contact (from Contacts)
	•	Tags / Status
	•	Actions: View / Edit / Delete

Click row → Party Detail screen.

⸻

1.2 Party Detail Screen

URL: /tenant/{tenantSlug}/parties/{partyId}

Sections:
	1.	Party Header
	•	Name, Type, Category
	•	Edit button (if TENANT_PARTY_MANAGE)
	•	Delete button (if TENANT_PARTY_MANAGE)
	2.	Addresses (tab or section)
	•	List of addresses for this party
	•	Add Address button (if manage)
	•	Each item: label, line1, city, country, “Primary” flag, Edit/Delete actions.
	3.	Contacts (tab or section)
	•	List of contacts for this party
	•	Add Contact button (if manage)
	•	Each item: name, role, email, phone, “Primary” flag, Edit/Delete actions.

Forms for Party / Address / Contact open as:
	•	Radix Dialog or Side Sheet (overlay form)
	•	On save, close & refresh detail / list

⸻

2. Flows & User Journeys (Tenant View)

F-TENANT-PARTY-01 – View Parties List

Actor: Tenant user with TENANT_PARTY_VIEW
Entry:
User is in tenant scope (/tenant/{slug}/dashboard) and clicks “Parties” in side nav.

Steps (UI + API):
	1.	Navigation
	•	/tenant/{slug}/parties route loads.
	•	Layout guard checks tenant access & permissions (SSR).
	2.	Fetch list
	•	Frontend calls:
	•	GET /api/v1/tenant/parties
(with paging / filter query params as you add them)
	3.	Render
	•	Show table of parties.
	•	If list is empty:
	•	Show empty state:
“No parties yet. Create the first party.”
	•	If user has TENANT_PARTY_MANAGE, show “Create party” CTA.
	4.	Permissions effect
	•	If user has only TENANT_PARTY_VIEW:
	•	New Party button hidden/disabled.
	•	Row click still navigates to details (read-only view).

If API returns 403 because permissions changed mid-session → show “You don’t have access” state and/or redirect to tenant dashboard.

⸻

F-TENANT-PARTY-02 – Create Party (Inline Form, Not Wizard)

Actor: Tenant user with TENANT_PARTY_MANAGE
Entry: Parties List → click New Party

Steps:
	1.	Open create form
	•	Click New Party.
	•	Open Radix Dialog (or right sheet) with form fields from Party (from tenant DB model), e.g.:
	•	Party Type (Person / Organization)
	•	Category (Vendor / Owner / Resident / Other)
	•	Name
	•	Code / ExternalRef (if exists)
	•	Tax/Government ID (optional)
	•	Notes
	2.	Submit
	•	On Save, client calls:
	•	POST /api/v1/tenant/parties
	•	Body: party payload.
	3.	Success
	•	API returns new partyId.
	•	UI:
	•	Close dialog.
	•	Option A (simple): Refresh list → highlight new row.
	•	Option B: Navigate to /tenant/{slug}/parties/{partyId}.
	4.	Validation / Errors
	•	Server validation errors → mapped to field errors (e.g. name required, duplicate code).
	•	401 / session expired → your global session dialog.
	•	403 → show “You no longer have permission to manage parties.”

If user doesn’t have TENANT_PARTY_MANAGE, backend returns 403 and UI shows “You don’t have permission” toast, even if button somehow appears.

⸻

F-TENANT-PARTY-03 – Edit Party

Actor: Tenant user with TENANT_PARTY_MANAGE
Entry: Parties list → click row → Party detail

Steps:
	1.	Open party detail
	•	GET /api/v1/tenant/parties/{id} to load full party entity.
	2.	Edit
	•	User clicks Edit on party header.
	•	Open same form as create, pre-filled.
	3.	Submit
	•	On Save, call:
	•	PUT /api/v1/tenant/parties/{id}
	4.	Success
	•	UI updates header & any list views.
	•	Show toast: “Party updated.”

If 404: show not found page for this party and provide link back to list.

⸻

F-TENANT-PARTY-04 – Add / Edit / Delete Party Addresses

04A – Add Address
Actor: Tenant user with TENANT_PARTY_MANAGE
Entry: Party detail → Addresses section

Steps:
	1.	Click Add Address → open address form:
	•	Label (Home, Office, Registered, etc.)
	•	Address lines, city, state, postcode, country
	•	“Primary” (yes/no)
	2.	Submit:
	•	POST /api/v1/tenant/parties/{partyId}/addresses
	3.	Success:
	•	Close dialog, refresh addresses list.
	•	If “Primary” checked:
	•	Backend enforces only one primary (might demote previous primary).

04B – Edit Address
	1.	Click Edit on an address row.
	2.	Load data locally (already from parent GET or re-fetch single address).
	3.	Submit:
	•	PUT /api/v1/tenant/parties/{partyId}/addresses/{id}
	4.	Success:
	•	Update list in UI.

04C – Delete Address
	1.	Click Delete on an address row → confirm dialog.
	2.	Submit:
	•	DELETE /api/v1/tenant/parties/{partyId}/addresses/{id}
	3.	On success:
	•	Remove row from list.

If user lacks TENANT_PARTY_MANAGE, the backend rejects all of these (403) even if buttons are visible.

⸻

F-TENANT-PARTY-05 – Add / Edit / Delete Contacts

Exactly symmetric to addresses but using contact endpoints.

05A – Add Contact
Actor: Tenant user with TENANT_PARTY_MANAGE
Steps:
	1.	On Contacts tab → click Add Contact.
	2.	Form fields (from PartyContact model):
	•	Name
	•	Role/Title (Owner, Manager, Accounts, etc.)
	•	Email
	•	Phone (mobile, landline)
	•	“Primary contact” toggle
	3.	Submit:
	•	POST /api/v1/tenant/parties/{partyId}/contacts
	4.	Success:
	•	Refresh contacts list.
	•	Backend enforces only one primary contact if that’s a rule.

05B – Edit Contact
	•	PUT /api/v1/tenant/parties/{partyId}/contacts/{id}

05C – Delete Contact
	•	DELETE /api/v1/tenant/parties/{partyId}/contacts/{id}

All governed by TENANT_PARTY_MANAGE.

⸻

F-TENANT-PARTY-06 – Delete Party

Actor: Tenant user with TENANT_PARTY_MANAGE
Entry: Party detail → Delete Party button (or from list row menu)

Steps:
	1.	User clicks Delete → confirm dialog:
	•	Warn if party is linked to leases / units / other domain entities (backend should enforce business rules).
	2.	On confirm:
	•	DELETE /api/v1/tenant/parties/{id}
	3.	Responses:
	•	200/204:
	•	Navigate back to /tenant/{slug}/parties
	•	Show toast “Party deleted.”
	•	409 Conflict if party has dependencies:
	•	UI shows message: “This party is in use (e.g. linked to leases or units) and can’t be deleted. Mark as inactive instead.”
(You can also add an IsActive toggle instead of hard delete.)

Backend still enforces referential rules based on tenant DB schema.  ￼

3. Authorization, Errors & Session Behaviour

3.1 Permissions in tenant shell
	•	Tenant layout uses your RequestContext:
	•	If user has no TENANT_PARTY_VIEW:
	•	“Parties” nav item is hidden.
	•	Direct navigation to /tenant/{slug}/parties* returns 403 → Unauthorized page.
	•	If user has view but not manage:
	•	They can see list & detail.
	•	“New”, “Edit”, “Delete”, “Add Address/Contact” actions hidden/disabled.

3.2 HTTP errors → UX
	•	401 from any of these APIs:
	•	Your global Session Expired dialog shows:
	•	“Continue session / Sign out”
	•	403:
	•	If from listing:
	•	Show “You don’t have access to parties in this community.”
	•	Optionally redirect to tenant dashboard.
	•	If from a single action (e.g. Save):
	•	Show toast: “You don’t have permission to perform this action.”
	•	404:
	•	GET /api/v1/tenant/parties/{id} not found:
	•	Show “Party not found” page with link back to list.
	•	Validation (4xx):
	•	Map to field errors on party/address/contact forms.
