🔷 Platform RBAC – Flows for Platform Admin

Scope: /platform/...
Side nav item: Team & Permissions → Roles, Users

Assume a platform permission like PLATFORM_RBAC_MANAGE is required to edit; view-only permission for read.

⸻

F-RBAC-PLAT-01 – View Platform Roles Overview

Actor: Platform Admin with PLATFORM_RBAC_VIEW
URL: /platform/rbac/roles

Goal: See all platform roles and how “powerful” each one is.

Steps:
	1.	User in platform scope opens Team & Permissions → Roles.
	2.	App calls:

GET /api/platform/rbac/roles

→ returns list of roles with:
	•	Role id, name, description
	•	PermissionCount
	•	UserCount (if you include it)

	3.	UI shows table:
	•	Columns: Role Name, Description, #Permissions, #Users
	•	Action per row: View role → role detail
	4.	If user has PLATFORM_RBAC_MANAGE:
	•	There’s a “View all permissions” link (optional) → uses GET /api/platform/rbac/permissions to show the full permission catalog.

⸻

F-RBAC-PLAT-02 – Edit Permissions for a Platform Role

Actor: Platform Admin with PLATFORM_RBAC_MANAGE
URL: /platform/rbac/roles/{roleId}

Goal: Enable/disable permissions for a specific platform role.

Steps:
	1.	From roles list, user clicks a role → /platform/rbac/roles/{id}.
	2.	App calls:

GET /api/platform/rbac/roles/{id}
GET /api/platform/rbac/permissions  // to show full list with grouping


	3.	Role Detail screen shows:
	•	Header: Role name, description, #users
	•	Body: Permissions matrix
	•	Grouped by module (e.g. Platform, TenantProvisioning, Billing, etc.)
	•	Each permission as a checkbox/toggle.
	4.	User toggles checkboxes:
	•	Internally tracked as a changed set.
	5.	Click “Save changes”:

PUT /api/platform/rbac/roles/{id}/permissions
// body: list of permission keys enabled for this role


	6.	On success:
	•	Show toast: “Role permissions updated”
	•	Update permission count on roles list (if cached)

Error cases:
	•	403 → show “You don’t have permission to manage platform roles.”
	•	409 → if backend prevents removing your own critical permissions, show message from server.

⸻

F-RBAC-PLAT-03 – View Users Assigned to a Platform Role

Actor: Platform Admin with PLATFORM_RBAC_VIEW
Entry: Role Detail screen

Goal: See who has this role.

Steps:
	1.	In /platform/rbac/roles/{id} there’s a “Users with this role” tab/section.
	2.	When opened:

GET /api/platform/rbac/roles/{id}/users


	3.	UI shows list:
	•	User name
	•	Email
	•	Status (Active/Disabled)
	•	Last login (if you have it)
	•	Action: Open user → /platform/rbac/users/{userId} (or reused users page filter)

View-only; role assignment changes done from Users screen.

⸻

F-RBAC-PLAT-04 – View Platform Users & Their Roles

Actor: Platform Admin with PLATFORM_RBAC_VIEW
URL: /platform/rbac/users (or /platform/users)

Goal: See all platform users and what roles they have.

Steps:
	1.	From side nav → Team & Permissions → Users.
	2.	App calls:

GET /api/platform/users

→ returns:
	•	User id, name, email
	•	List of role names / ids

	3.	UI shows table:
	•	Name, Email, Roles (chips/badges), Status
	•	Actions:
	•	“Manage roles” (if manage permission)
	4.	Filters:
	•	Search by name/email
	•	Filter by role

⸻

F-RBAC-PLAT-05 – Assign / Remove Roles for a Platform User

Actor: Platform Admin with PLATFORM_RBAC_MANAGE
Entry: Platform Users list → click “Manage roles”

Steps:
	1.	In /platform/rbac/users, user clicks “Manage roles” for a given user.
	2.	Dialog opens:
	•	Shows all available roles (from roles endpoint or cached).
	•	Current user roles pre-checked.
	3.	Admin checks/unchecks roles as needed.
	4.	Click Save:

PUT /api/platform/users/{id}/roles
// body: list of role ids to assign


	5.	On success:
	•	Dialog closes.
	•	Users table row updates (role chips).
	•	Toast: “User roles updated.”

Safety:
	•	Backend should prevent assigning/removing roles that could lock out the last PlatformAdmin; if so, return validation error → UI shows message.

🔶 Tenant RBAC – Flows for Community Admin

Scope: /tenant/{tenantSlug}/...
Side nav item: Team & Access → Roles, Users

Mirrors the platform RBAC, but tenant-scoped.

⸻

F-RBAC-TENANT-01 – View Tenant Role Groups (Tenant RBAC Roles)

Actor: Community Admin with TENANT_RBAC_VIEW
URL: /tenant/{slug}/rbac/roles

Goal: See all tenant role groups (e.g., CommunityAdmin, Manager, Security, Accountant).

Steps:
	1.	In tenant scope, user clicks Team & Access → Roles.
	2.	App calls:

GET /api/tenant/rbac/roles


	3.	UI shows list:
	•	RoleGroup Name
	•	Description
	•	#Permissions, #Users
	4.	Clicking a role group opens its detail view.

⸻

F-RBAC-TENANT-02 – Edit Permissions for a Tenant Role Group

Actor: Community Admin with TENANT_RBAC_MANAGE
URL: /tenant/{slug}/rbac/roles/{id}

Goal: Control which tenant-level features each role group can access.

Steps:
	1.	From tenant roles list, click role → /tenant/{slug}/rbac/roles/{id}.
	2.	App calls:

GET /api/tenant/rbac/roles/{id}
GET /api/tenant/rbac/permissions


	3.	Screen shows:
	•	Role group header: name, description, #users.
	•	Permission matrix:
	•	Grouped by modules: Units, Parties, Ownership, Parking, Maintenance, Visitors, etc.
	•	Each permission a checkbox: TENANT_UNIT_VIEW, TENANT_UNIT_MANAGE, etc.
	4.	Admin toggles checkboxes.
	5.	Click Save:

PUT /api/tenant/rbac/roles/{id}/permissions


	6.	On success:
	•	Toast: “Role group permissions updated”
	•	Count updated in roles list.

Errors:
	•	403 if user lacks manage permission → show unauthorized state.

F-RBAC-TENANT-05 – Assign / Remove Role Groups for a Community User

Actor: Community Admin with TENANT_RBAC_MANAGE
Entry: Tenant Users list → “Manage roles”

Steps:
	1.	On /tenant/{slug}/rbac/users, click “Manage roles” for a user.
	2.	Dialog:
	•	Shows all tenant role groups.
	•	Current user role groups pre-checked.
	3.	Admin adjusts selections.
	4.	Click Save:

PUT /api/tenant/users/{id}/roles
// body: list of tenant role group ids


	5.	On success:
	•	Dialog closes.
	•	Role chips refresh in the list row.
	•	Toast: “Community user roles updated.”

Safety:
	•	Backend should prevent removing the last Community Admin from a tenant.
	•	If that happens, API returns validation error → show friendly message:
“Each community must have at least one Community Admin.”

🔁 Shared UX ideas (Platform & Tenant RBAC)
	•	Permission catalog screens (optional):
	•	/platform/rbac/permissions
	•	/tenant/rbac/permissions
	•	Purely read-only list grouped by feature.
	•	Read vs Manage modes:
	•	If user has only *_RBAC_VIEW:
	•	They can see roles/users but not toggle permissions.
	•	Checkboxes disabled, no save button.
	•	404 / Unauthorized / Session:
	•	401 → your existing session expiry dialog.
	•	403 from any RBAC endpoint:
	•	Show “You don’t have access to manage roles & permissions in this scope.”
	•	404 → if roleId/userId not valid, show scoped Not Found page.
